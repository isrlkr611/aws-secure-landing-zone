# Compliance Mapping (Brique 2)

Answers a different question than `tfsec`/`checkov` in CI do. They ask "did this change introduce a vulnerability" and block the merge if so. This module asks "**which compliance framework clauses does our current, already-passing infrastructure actually satisfy**", and reports a percentage per framework, computed from real, currently-passing Checkov results - not a self-assessment questionnaire, not a description of intent.

This is the same category of feature as Vanta/Drata/Secureframe (continuous-compliance platforms that turn technical scan results into audit-ready evidence), scoped down to the one input this repo already produces real data for: Terraform/Checkov findings on `terraform/`.

## How it works

1. `checkov -d terraform/ --output json` runs for real (not simulated) and produces structured pass/fail/skip results per check, per resource.
2. [`mapping_norms.yaml`](mapping_norms.yaml) groups the 61 distinct Checkov check IDs that actually appear in this repo's scan into 6 control themes (network exposure, encryption at rest, IAM least privilege, authentication/secrets, logging/monitoring, config hygiene), each mapped to specific clauses in **ISO/IEC 27001:2022** (Annex A), **PCI-DSS v4.0**, the **NIS2 Directive** (Art. 21(2)), and **SOC 2** (Trust Services Criteria).
3. [`generate_report.py`](generate_report.py) cross-references the two and computes, per framework: how many mapped checks were evaluated, how many passed, how many failed, and a compliance percentage.
4. Output: [`reports/compliance-report.md`](reports/compliance-report.md) (human-readable) and `reports/compliance-report.json` (machine-readable), both regenerated on every CI run of `.github/workflows/terraform-security-scan.yml` and published as a build artifact + job summary.

## Run it yourself

```bash
pip install -r compliance/requirements.txt
python3 compliance/generate_report.py \
  --output-md compliance/reports/compliance-report.md \
  --output-json compliance/reports/compliance-report.json
```

Or point it at an already-generated Checkov JSON report instead of re-running checkov:

```bash
checkov -d terraform/ --output json > /tmp/checkov.json   # NOT --quiet: see the comment in generate_report.py
python3 compliance/generate_report.py --checkov-json /tmp/checkov.json
```

## Methodology, precisely

- **Only check IDs that actually show up when scanning this repo's `terraform/` tree are mapped.** This is not a generic "every AWS Checkov check ever" reference table - see the header of `mapping_norms.yaml`. Any check_id the scan encounters that isn't in the mapping file is listed in the report's "Unmapped checks" section rather than silently dropped, so this file's coverage gaps stay visible.
- **Compliance % = passed / (passed + failed) among checks mapped to that framework.** Checks with a documented `checkov:skip=...` exception (see `docs/architecture.md` "Security Choices" for the reasoning behind each one currently in this repo) are counted separately as "documented exceptions requiring compensating-control review" and are **excluded from the denominator** - an accepted risk is not the same thing as a satisfied control, and folding the two together would overstate compliance to make the number look better. A real auditor would still need to review each documented exception individually; this report surfaces exactly which ones and why, rather than hiding them inside a headline percentage.
- **A theme can map to more than one framework, and a framework's score is the union of every theme mapped to it** - e.g. IAM least privilege alone contributes 22 of the 61 tracked checks and touches all four frameworks, so it dominates each framework's denominator. This is intentional: least-privilege IAM genuinely is the largest single control category across all four frameworks tracked here.
- **Framework clause references were chosen conservatively.** Where a Checkov check theme maps clearly to a named requirement (e.g. "no SSH from 0.0.0.0/0" → PCI-DSS Requirement 1, network security controls), it's included. Where a theme is really about implementation hygiene rather than a specific compliance clause (see `config_hygiene` in `mapping_norms.yaml` - e.g. "every security group has a description"), it is mapped to fewer frameworks rather than force-fitting a clause reference that wouldn't survive an auditor's scrutiny.

## What this is not

- **Not a certification, not an audit, not legal or compliance advice.** It's a technical control inventory cross-referenced against public framework text, generated by re-running the same static analyzer already used elsewhere in this repo. An ISO 27001 or PCI-DSS assessment covers organizational and process controls (policies, training, vendor management, physical security, incident response procedures) that no Terraform scanner can see.
- **Not exhaustive per framework.** ISO 27001 Annex A alone has 93 controls; this maps the ~15 that this specific infrastructure's Checkov findings speak to. A 100% score here means "the infrastructure controls we scan for are in place," not "this organization is ISO 27001 certifiable."
- **Not yet wired to Brique 1 (`platform/`).** The original product brief describes reusing findings from a cloud posture scanner *and* the attack-surface-monitor's own findings (new open ports, expiring certs, leaked credentials). Only the Terraform/Checkov side is implemented so far - see the root README's roadmap section.
